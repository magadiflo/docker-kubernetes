## [Docker Compose overview](https://docs.docker.com/compose/)

`Compose` es una herramienta para definir y ejecutar aplicaciones `Docker multicontenedor`. Con `Compose`, utilizas un
archivo `YAML` para configurar los servicios de tu aplicaciĂłn. Luego, con un solo comando, creas e inicias todos los
servicios a partir de tu configuraciĂłn.

`Compose` funciona en todos los entornos; `producciĂłn`, `staging`, `desarrollo`, `pruebas`, asĂ­ como flujos de trabajo
CI. TambiĂ©n dispone de comandos para gestionar todo el ciclo de vida de tu aplicaciĂłn:

- Iniciar, detener y reconstruir servicios
- Ver el estado de los servicios en ejecuciĂłn
- Transmitir la salida de registro de los servicios en ejecuciĂłn
- Ejecutar un comando puntual en un servicio

Las caracterĂ­sticas clave de `Compose` que lo hacen eficaz son:

- Disponer de mĂşltiples entornos aislados en un Ăşnico host
- Conservar los datos de volumen cuando se crean contenedores
- Recrear sĂłlo los contenedores que han cambiado
- Soportar variables y mover una composiciĂłn entre entornos

### [The Compose file](https://docs.docker.com/compose/intro/compose-application-model/#the-compose-file)

Antes de empezar a crear el archivo `compose.yml` (en versiones anteriores era `docker-compose.yml`) veamos algunos
aspectos sobre Ă©l.

El archivo `Compose` es un archivo `YAML` que define:

- Version `(Optional)`
- Services `(Required)`
- Networks
- Volumes
- Configs
- Secrets

La ruta predeterminada para un archivo `Compose` es `compose.yaml` (preferido) o `compose.yml` que se coloca en el
directorio de trabajo. `Compose` tambiĂ©n admite `docker-compose.yaml` y `docker-compose.yml` para la compatibilidad
con versiones anteriores. Si existen ambos archivos, `Compose` prefiere el canĂłnico `compose.yaml`.

### [Elemento de nivel superior VersiĂłn](https://docs.docker.com/compose/compose-file/04-version-and-name/)

La propiedad de nivel superior `versiĂłn` estĂˇ definida por la EspecificaciĂłn Compose para compatibilidad con versiones
anteriores. `Solo tiene carĂˇcter informativo`.

**Compose no utiliza la versiĂłn para seleccionar un esquema exacto para validar el archivo Compose, sino que prefiere
el esquema mĂˇs reciente cuando estĂˇ implementado.**

### [Elemento de nivel superior de los Servicios](https://docs.docker.com/compose/compose-file/05-services/)

Un `servicio `es una definiciĂłn abstracta de un recurso informĂˇtico dentro de una aplicaciĂłn que **puede escalarse o
sustituirse independientemente de otros componentes**.
`Los servicios estĂˇn respaldados por un conjunto de contenedores`, ejecutados por la plataforma de acuerdo con
los requisitos de replicaciĂłn y las restricciones de ubicaciĂłn. **Dado que los servicios estĂˇn respaldados por
contenedores, se definen mediante una imagen Docker y un conjunto de argumentos de tiempo de ejecuciĂłn.**
`Todos los contenedores de un servicio se crean de forma idĂ©ntica con estos argumentos.`

Un archivo Compose debe declarar un elemento de nivel superior de servicios como un mapa cuyas `claves` **son
representaciones de cadenas de nombres de servicios**, y cuyos `valores` **son definiciones de servicios.**
Una `definiciĂłn de servicio` **contiene la configuraciĂłn que se aplica a cada contenedor de servicio.**

**Cada servicio tambiĂ©n puede incluir una secciĂłn de** `build`, **que define cĂłmo crear la imagen Docker para el
servicio.** `Compose permite crear imĂˇgenes Docker utilizando esta definiciĂłn de servicio.` Si no se utiliza,
la secciĂłn de construcciĂłn se ignora y el archivo Compose sigue considerĂˇndose vĂˇlido.

**Cada servicio define restricciones y requisitos de tiempo de ejecuciĂłn para ejecutar sus contenedores.** La secciĂłn
de `deploy` agrupa estas restricciones y permite a la plataforma ajustar la estrategia de despliegue para adaptar mejor
las necesidades de los contenedores a los recursos disponibles. Si no se implementa, la secciĂłn de despliegue se ignora
y el archivo Compose sigue considerĂˇndose vĂˇlido.

A continuaciĂłn se muestran algunos atributos usados dentro de un servicio:

- `build`, especifica la configuraciĂłn de compilaciĂłn para crear una imagen de contenedor a partir del cĂłdigo fuente,
  tal y como se define en la especificaciĂłn de compilaciĂłn de Compose.


- `container_name`, es una cadena que especifica un nombre de contenedor personalizado, **en lugar de
  un nombre generado por defecto.**


- `volumes`, definen rutas de host de montaje o `volĂşmenes con nombre` que son accesibles por contenedores
  de servicio. Puedes usar volĂşmenes para definir mĂşltiples tipos de montajes; `volumen`, `bind`, `tmpfs` o `npipe`.

  > Para `reutilizar un volumen a travĂ©s de mĂşltiples servicios`, se debe declarar un `volumen con nombre` **en la clave
  > de volĂşmenes de nivel superior.**
  >
  > `La declaraciĂłn de volĂşmenes de nivel superior` permite **configurar volĂşmenes con nombre que pueden reutilizarse en
  > varios servicios.** Para utilizar un volumen en varios servicios, debe conceder explĂ­citamente acceso a cada
  > servicio mediante el atributo volumes.

- `networks`, capa que permite a los servicios comunicarse entre sĂ­. `El elemento de nivel superior networks permite
  configurar redes con nombre que pueden reutilizarse en varios servicios`. Para utilizar una red en varios servicios,
  debes conceder explĂ­citamente acceso a `cada servicio` utilizando el atributo `networks`.

  > PodrĂ­amos no configurar explĂ­citamente un `networks` y en ese caso, por defecto, `Compose` configura una Ăşnica red
  > para tu aplicaciĂłn. Cada contenedor para un servicio se une a la red por defecto y es accesible por otros
  > contenedores en esa red, y detectable por ellos en un nombre de host idĂ©ntico al nombre del contenedor.

  `ÂżQuĂ© es un healthcheck?`.

El `healthcheck` es una funcionalidad en `Docker` que permite monitorear el estado de salud de un contenedor. Define una
serie de condiciones que `Docker` verificarĂˇ periĂłdicamente para determinar si el contenedor estĂˇ funcionando
correctamente. Si el contenedor no pasa las pruebas de salud, se puede tomar acciĂłn, como reiniciar el contenedor o
marcarlo como `no saludable`.

Tomemos como ejemplo el `healthcheck` del servicio de `postgres`.

````yml

  timeout: 5s
  retries: 5
````

1. `test: [ 'CMD-SHELL', 'pg_isready -U $${POSTGRES_USER} -d $${POSTGRES_DB}' ]`
    - Este es el comando que `Docker` ejecuta para verificar si `PostgreSQL` estĂˇ disponible y listo para aceptar
      conexiones.
    - `CMD-SHELL`: Indica que el comando serĂˇ ejecutado en un `shell`, lo que permite usar el comando como si lo
      estuvieras ejecutando en la terminal.
    - `pg_isready`: Es una herramienta de `PostgreSQL` que se utiliza especĂ­ficamente para verificar si el servidor de
      base de datos estĂˇ disponible y listo para aceptar conexiones. No inicia una conexiĂłn real, pero revisa el estado
      del servidor.
    - `-U $${POSTGRES_USER}`: Especifica el usuario que se usarĂˇ para realizar la comprobaciĂłn.
    - `-d $${POSTGRES_DB}`: Indica la base de datos sobre la cual se estĂˇ verificando la disponibilidad.

   En resumen, este comando verifica que el servidor `PostgreSQL` estĂ© corriendo y sea capaz de aceptar conexiones para
   el usuario `postgres` en la base de datos `db_course_service`.


2. `interval: 10s`, este parĂˇmetro define el intervalo de tiempo entre cada ejecuciĂłn del `healthcheck`, es decir, cada
   cuĂˇnto tiempo `Docker` ejecutarĂˇ el comando de prueba. En este caso, el intervalo es de 10 segundos. `Docker`
   ejecutarĂˇ la prueba cada 10 segundos mientras el contenedor estĂ© corriendo.


3. `timeout: 5s`, este parĂˇmetro establece el tiempo mĂˇximo que `Docker` esperarĂˇ a que la prueba de `healthcheck` se
   complete antes de considerarla fallida. AquĂ­ se ha configurado un timeout de 5 segundos. Si el comando no devuelve
   una respuesta en ese tiempo, se considera que la prueba ha fallado.


4. `retries: 5`, este parĂˇmetro define cuĂˇntas veces `Docker` intentarĂˇ ejecutar el `healthcheck` antes de marcar el
   contenedor como `no saludable`. En este caso, si la prueba falla 5 veces consecutivas, `Docker` marcarĂˇ el contenedor
   como no saludable.

### Sobre el $${POSTGRES_USER}

En `Docker Compose` no existe un soporte directo para interpolaciĂłn de variables de entorno dentro de los comandos de
`healthcheck`. Sin embargo, podemos solucionarlo utilizando variables de entorno dentro del contenedor para evitar
repetir valores.

Una forma es aprovechar que los valores de las variables de entorno definidos en el servicio (como `POSTGRES_USER` y
`POSTGRES_DB`) estĂˇn disponibles dentro del contenedor, y podemos utilizarlas en el comando `test` del `healthcheck`.
Lo Ăşnico que necesitamos es acceder a esas variables usando la siguiente sintaxis `$${POSTGRES_USER}`.

La sintaxis con doble sĂ­mbolo de dĂłlar (`$${POSTGRES_USER}`)  generalmente se usa en `Docker Compose` para evitar que
`Docker Compose` intente interpretar y expandir la variable cuando estĂˇ creando el contenedor. Esto es Ăştil cuando
deseas que la variable de entorno se expanda dentro del contenedor, no en el archivo `compose.yml` durante la creaciĂłn
del contenedor.

Al usar `$${POSTGRES_USER}`, le decimos a `Docker` que no expanda la variable de entorno `POSTGRES_USER` en ese
momento, sino que pase el nombre de la variable tal cual al contenedor, para que este la resuelva cuando se ejecute
el comando en su entorno interno.

En resumen del `$${POSTGRES_USER}`, el doble sĂ­mbolo de dĂłlar `($$)` es un truco para evitar que `Docker Compose`
expanda la variable en el momento de la compilaciĂłn del `compose.yml`. En su lugar, pasa literalmente el texto
`POSTGRES_USER` al contenedor, que luego lo resolverĂˇ usando las variables de entorno internas cuando se ejecute.

**Nota 1**
> Cuando hablo de `expandir la variable` en el contexto de `Docker Compose` (y, en general, en programaciĂłn
> y scripting), me refiero al proceso de reemplazar el nombre de la variable con su valor correspondiente. Es decir,
> la expansiĂłn es el acto de tomar una variable definida (como `POSTGRES_USER`) y sustituirla por su valor
> (por ejemplo, `postgres`) en el momento en que se procesa el archivo.

**Nota 2**
